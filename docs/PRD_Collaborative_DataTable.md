# 产品需求文档 (PRD): 协同 DataTable React 组件库

## 1. 背景与业务目标
当前业务系统在处理大规模研发数据展示时存在渲染性能瓶颈，且在多人在线协作场景下，容易出现数据覆盖和状态不一致的问题。
本项目目标是开发一套 **开箱即用、支持 5000 行级数据、具备实时协同能力的 DataTable React 组件库**，提供流畅的"查看-编辑-同步"一体化体验。

*(注：项目包含一个独立的 Demo 后端沙箱用于功能验证。文档后续将明确区分组件库与 Demo 环境的职责边界。)*

## 2. 核心用户
* **主要触达受众**: **前端开发工程师**，负责将表格组件集成到业务系统中。

---

## 3. 组件功能需求 (Component Core)

### 3.1 高性能渲染
*   **虚拟滚动**: 支持 5000 行级数据的流畅滚动，首屏加载时间 ≤ 300ms，滚动帧率保持在 60fps 左右。

### 3.2 列定义与类型系统 *(新增)*
*   **类型化列定义**: 组件通过 Props 接收外部传入的列配置数组（`ColumnDef[]`），每列需声明数据类型（`string` / `number` / `enum` / `date`）。
*   **不允许写死列**: 列定义必须完全由外部宿主配置，组件内部不预设任何业务相关的列。
*   **类型驱动行为**: 列类型将自动影响以下行为：
    * 排序逻辑（数值排序 vs 字典序排序）
    * 筛选控件（文本输入 / 数值范围 / 枚举多选）
    * 编辑控件（Input / NumberInput / Select / DatePicker）

### 3.3 本地视图操作 *(增强)*
*   **排序**: 支持列排序，仅在本地生效，不触发协同同步。
*   **高级筛选**:
    * P0: 支持按单字段筛选，根据列类型自动适配条件控件：
        * `string` → 包含/不包含
        * `number` → 大于/小于/范围
        * `enum` → 多选包含
    * P1: 支持多条件筛选（AND/OR 组合）
*   **分组 (Group)**:
    * 分组控件移至顶部工具栏，以下拉选择器形式呈现，最多支持两层嵌套分组。
    * 分组视觉风格采用卡片式布局（参考飞书多维表格），每个分组显示为带标题的独立区块，内部保持标准表格列结构。

### 3.4 协作式编辑 *(增强)*
*   **类型感知编辑**: 根据列类型自动选择对应的编辑控件：
    * `string` → `<input type="text">`
    * `number` → `<input type="number">`
    * `enum` → `<Select>` 下拉，选项由列定义中的 `enumOptions` 提供
    * `date` → DatePicker
*   **原子化更新**: 修改操作直接作用于 Yjs 的数据节点，通过引用追踪确保并发安全。
*   **自定义操作扩展**: 提供 `cellEditor` RenderProps 插槽，支持弹出 Modal 进行复杂计算。

### 3.5 实时状态感知 (Awareness)
*   **协同光标**: 实时展示其他协作者正在选中的单元格。
*   **身份标识**: 在选中的单元格周围显示特定颜色的边框及人员姓名标签。
*   **状态清理**: 当用户断开连接时，对应的光标状态需在 60 秒内自动清理。

## 4. Demo 辅助环境需求 (Demo App Sandbox)

### 4.1 数据网关
*   **WebSocket 服务**: 负责接收并转发来自各端的 Yjs 更新切片。
*   **房间管理**: 支持多房间隔离的实时协同。

### 4.2 数据持久化
*   **快照备份**: 服务器每隔 1 分钟自动将内存中的 Y.Doc 持久化为二进制快照到数据库（SQLite），防止因服务意外重启导致的数据丢失。
*   **定时归档**: 当房间闲置超过 5 分钟，自动将二进制镜像转化为 JSON 格式存储到业务记录表中。
*   **测试数据生成**: 提供初始化脚本，自动填充 5000 条结构化测试数据。

---

## 5. 工作流描述
1.  **挂载启动**: 宿主系统传入 `columns` 列定义和 `roomId`，DataTable 组件建立 WebSocket 连接并同步首屏数据。
2.  **视角切换**: 用户在工具栏选择分组字段（最多 2 层）、设置筛选条件或点击列头排序。这些操作仅改变当前用户的本地视图 State。
3.  **协同编辑**: 用户双击单元格，组件根据列类型渲染对应编辑控件。修改确认后生成增量 patch 发送至网关。
4.  **状态感知**: 用户光标坐标通过 Awareness 协议广播给同一房间的其他用户。

## 6. 数据模型 (Demo 环境)
| 字段 | 类型 | 说明 |
|---|---|---|
| `room_id` | uuid | 协同房间 ID |
| `row_id` | uuid | 数据行唯一标识 |
| `data` | jsonb | 结构化业务数据内容 |
| `snapshot` | bytea | Y.Doc 二进制快照（热备份） |

---

## 7. 风险评估
*   **数据库性能压力**: 高频的快照备份在房间数量极多时可能产生 IO 压力。
*   **分组渲染复杂度**: 卡片式分组 + 虚拟滚动的结合需要额外处理动态区块高度和嵌套滚动。
